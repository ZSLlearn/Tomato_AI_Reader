# 多格式AI阅读器PRD（聚焦流程+界面+交互）
## 一、产品概述
### 1.1 产品定位
一款采用IDE式界面的多格式智能阅读器，核心聚焦「文档管理流程简化+阅读与AI对话的高效交互」，支持主流文档/电子书原生预览，通过「左侧资源管理+中间阅读+右侧AI对话」一体化布局，满足用户快速管理文档、精准阅读、基于全文的AI互动需求。

### 1.2 核心目标
- 简化文档导入、书架管理的操作流程，降低使用门槛；
- 打造直观的IDE式界面交互，边栏折叠/展开、阅读操作、AI提问流程连贯；
- 实现「文档-对话」强绑定，选中文字/手动提问均能基于全文上下文快速响应，交互反馈清晰。

### 1.3 目标用户
学生、职场人士、研究者等需阅读多格式文档（PDF/PPT/DOCX/EPUB等），并希望通过AI辅助理解内容的用户。

## 三、详细模块设计（聚焦流程与交互）
### 3.1 文档管理模块（核心：流程简化+交互直观）
#### 3.1.1 书架管理流程
1. 初始界面：默认显示「全部文档」「最近阅读」两个系统书架，置顶展示；
2. 新建书架：点击界面顶部「+」按钮→输入书架名称→确认创建，书架按创建时间排序；
3. 书架操作：右键点击书架→选择「重命名」/「删除」，删除时弹窗提示「是否删除该书架？文档将保留在全部文档」；
4. 文档移动：
   - 单个移动：右键文档→「移动到」→选择目标书架→即时刷新；
   - 批量移动：按住Ctrl选中文档→右键→「批量移动」→选择书架→完成。

#### 3.1.2 文档导入流程
1. 导入入口：界面顶部「导入文档」按钮+文档列表区域拖拽入口（显示「拖拽文件到此处导入」提示）；
2. 操作流程：
   - 本地导入：点击「导入文档」→打开文件选择器→勾选单个/多个文件→「确认导入」；
   - 拖拽导入：选中桌面/文件夹文件→拖拽至文档列表→松开鼠标→自动开始导入；
3. 反馈交互：
   - 导入中：显示进度条（标注「导入中：X/Y个文件」）；
   - 成功后：文档列表即时刷新，新导入文档置顶显示，弹窗提示「X个文件导入成功」（3秒自动消失）；
   - 失败时：弹窗提示「以下文件导入失败：XXX（格式不支持/文件损坏）」，提供「查看支持格式」入口。

#### 3.1.3 文档操作交互
1. 排序交互：点击文档列表顶部「名称」/「导入时间」/「大小」→切换升序/降序（点击时显示箭头标识排序方向）；
2. 搜索交互：点击搜索框→输入关键词→实时过滤文档（匹配文件名/文档预览内容）→关键词高亮显示，无结果时提示「未找到相关文档」；
3. 右键菜单：选中文档→右键弹出菜单→包含「打开」「重命名」「移动到」「删除」「查看属性」，点击对应选项即时响应。

### 3.2 阅读界面模块（核心：IDE式布局+流畅操作）
#### 3.2.1 整体布局交互
- 界面结构：左侧（资源管理器）+中间（阅读区）+右侧（AI对话区），三者宽度均可通过拖拽边缘调整；
- 边栏切换：
  - 左侧边栏：点击顶部「←」按钮→折叠（仅显示目录图标），点击图标→展开；
  - 右侧边栏：点击顶部「→」按钮→隐藏（仅显示「AI对话」入口），点击入口→唤起；
- 窗口适配：调整软件窗口大小时，中间阅读区优先适配，左右边栏保持当前宽度比例，最小窗口时自动折叠边栏，保留阅读区核心显示。

#### 3.2.2 左侧资源管理器交互
1. 目录加载：打开文档后→自动解析目录→1秒内显示（大文档显示「加载目录中...」）；
2. 目录交互：
   - 折叠/展开：点击目录前「+」（展开）/「-」（折叠）图标，支持「一键全部展开/折叠」（顶部小按钮）；
   - 跳转操作：点击目录项→中间阅读区即时跳转到对应页面，同时目录项高亮显示（同步当前阅读位置）；
   - 无目录处理：显示「未识别到目录」，提供「生成目录」按钮→点击后基于文档标题自动提取目录。

#### 3.2.3 中间阅读区核心交互（原生预览+基础操作）
1. 基础操作流程：
   - 缩放：点击阅读区顶部「+」「-」按钮/快捷键→实时缩放，「适合宽度」「适合页面」按钮→一键适配；
   - 翻页：PDF/PPT类文档→点击左右页边距/快捷键←→/鼠标滚轮→单页翻；双页模式→左右同步滚动；
   - 文字选中：鼠标拖拽选中任意文字→选中区域高亮（默认黄色，可在设置中切换）；
2. 选中后交互：
   - 右键菜单：选中后右键→「复制」「AI提问」「标注」（可选）；
   - 拖拽操作：按住选中文字→拖拽至右侧AI对话区→松开鼠标→自动生成提问（如「解释这段内容」）；
3. 阅读模式切换：
   - 单页/双页：点击顶部「单页」「双页」按钮→即时切换（双页模式下左右分栏显示）；
   - 夜间模式：点击顶部「月亮」图标→阅读区背景变黑、文字变白（不影响文档图片/表格色彩）。

### 3.3 AI对话模块（核心：流程连贯+交互清晰）
#### 3.3.1 对话绑定基础流程
1. 打开文档：右侧对话区自动加载该文档的「最近一次对话」，无对话时显示「欢迎提问！可选中文字或手动输入」；
2. 切换文档：对话区自动清空当前内容，加载新文档的对话历史（按时间倒序）；
3. 删除文档：弹窗提示「是否同时删除该文档的所有对话？」→确认后同步删除，取消则仅删除文档，对话保留（标注「关联文档已删除」）。

#### 3.3.2 对话操作交互
1. 新建对话：点击对话区顶部「+ 新建对话」→清空当前对话内容→生成新对话标识（如「对话1」「对话2」），历史对话保留在下拉列表；
2. 历史对话：点击对话区顶部下拉框→显示所有历史对话（按时间倒序，标注创建时间）→点击任意对话→加载该对话的「提问+回答」内容；
3. 删除对话：选中某条历史对话→点击「×」按钮→弹窗提示「是否删除该对话？」→确认后即时删除，不可恢复。

#### 3.3.3 AI提问核心流程（3种方式，聚焦交互）
##### 方式1：选中文字右键提问（高频）
1. 阅读区选中文字→右键→点击「AI提问」；
2. 对话区自动填充提问模板（如「请解释这段文字：{选中内容}」），光标定位在模板后，支持用户补充提问；
3. 点击「发送」按钮/按Enter→对话区显示「AI思考中...」动画；
4. 1-5秒后显示AI回答，回答开头标注「基于文档第X章/第X页」。

##### 方式2：拖拽选中文字提问
1. 阅读区选中文字→按住选中区域→拖拽至对话区输入框；
2. 输入框自动填充选中内容，并生成默认提问（如「分析这段内容的核心观点」）；
3. 用户可修改提问内容→点击「发送」→获取AI回答。

##### 方式3：手动输入提问
1. 对话区输入框手动输入问题（如「文档的核心结论是什么？」「第2章和第5章有什么关联？」）；
2. 点击「发送」/按Enter→显示加载动画→AI基于全文内容回答，标注关联位置。

#### 3.3.4 上下文交互反馈
- 回答关联：无论哪种提问方式，AI均基于整篇文档内容回答，避免仅局限于选中文字/当前页；
- 连续提问：同一对话中，后续提问自动关联前序对话内容（如先问「什么是A？」，再问「它的应用场景是什么？」，AI自动关联「A的应用场景」）；
- 无相关内容：AI回答「未在文档中找到相关内容，是否基于通用知识补充？」，提供「是」「否」两个按钮，点击「是」则基于通用知识回答，点击「否」则结束响应。

### 3.4 基础设置模块（简化，仅保留核心交互）
1. 主题切换：界面顶部「设置」→「主题」→选择「明色」「暗色」「跟随系统」→即时生效；
2. 选中高亮：「设置」→「选中颜色」→选择黄/绿/蓝3种默认颜色→阅读区选中效果即时预览；
3. 回答长度：「设置」→「AI回答长度」→选择「短（≤100字）」「中（≤300字）」「长（≤500字）」→后续提问自动适配。

### 4.2 反馈交互设计（让用户感知操作结果）
- 操作反馈：点击按钮/菜单时，按钮变色（hover效果），点击后有轻微震动反馈（可选）；
- 加载反馈：文档导入、目录解析、AI提问时，显示转圈动画+文字提示（如「导入中...」「思考中...」）；
- 成功/失败反馈：导入成功、对话发送成功时，显示绿色小对勾（停留2秒）；失败时显示红色叉号+文字提示；
- 滚动反馈：阅读区滚动时，左侧目录同步高亮当前章节；目录滚动时，阅读区同步跳转对应位置。

### 4.3 界面一致性规范
- 按钮样式：所有功能按钮（导入、新建、发送等）采用统一圆角设计，颜色一致（主色调），hover时亮度变化；
- 弹窗样式：所有确认弹窗（删除、导入失败等）结构一致（标题+内容+确认/取消按钮），位置居中；
- 文字样式：同一模块文字大小、颜色统一，重要操作（如「AI提问」）文字加粗或标红，突出显示。

## 五、用户场景示例（聚焦操作流程）
### 场景1：学生阅读PDF论文并提问
1. 学生打开软件，点击「导入文档」→选择电脑中的PDF论文→导入成功，论文显示在「全部文档」顶部；
2. 点击论文名称→打开阅读界面，左侧显示论文目录，中间还原论文排版，右侧显示空白对话区；
3. 学生点击目录「3. 实验结果」→阅读区跳转到对应页面，拖拽选中某段实验结论文字；
4. 按快捷键Ctrl+Q→对话区自动填充「解释这段文字：{选中内容}」，学生补充「结合全文说明该结果的意义」；
5. 点击「发送」→显示「AI思考中...」→几秒后AI生成回答，标注「基于论文第3章实验结果+第5章结论分析」；
6. 学生继续提问「这个实验方法有什么局限性？」→AI基于全文相关章节（如「4. 讨论」）补充回答。

### 场景2：职场人管理PPT并AI分析
1. 职场人将电脑中的PPT文件拖拽至软件文档列表→显示导入进度条→提示「1个文件导入成功」；
2. 右键点击PPT→选择「移动到」→新建「工作汇报」书架→PPT成功移动至该书架；
3. 打开PPT→左侧显示幻灯片目录（按页码+标题），中间按单页模式显示幻灯片，完整还原排版；
4. 选中幻灯片中「2024年营收数据」表格→拖拽至右侧对话区→自动生成「分析这份营收数据的变化趋势」；
5. 点击发送→AI基于PPT中所有数据页（如往期营收对比、市场分析页）生成趋势分析，标注「基于幻灯片5、8、12页数据」；
6. 点击对话区「+ 新建对话」→输入「总结这份PPT的核心建议」→AI基于全文生成3条核心建议。

## 六、AI功能体系规划（三层架构）

### 6.1 设计理念：为什么需要三层架构？

AI阅读器的核心矛盾在于：**用户既需要快速获取信息（不离开阅读上下文），又需要进行深度分析（需要对话式交互）**。将所有AI功能全部塞进右侧对话面板会导致高频轻量操作路径过长，而全部放在悬浮窗又无法承载复杂分析。因此采用三层架构，按「响应速度 × 交互深度」两个维度划分功能。

### 6.2 三层架构总览

```
┌─────────────────────────────────────────────────────────┐
│                    AI 功能三层架构                        │
├─────────────┬───────────────────┬───────────────────────┤
│   L1 即时层  │    L2 对话层       │     L3 异步层          │
│  （悬浮窗）   │  （右侧AI面板）     │   （后台任务）          │
├─────────────┼───────────────────┼───────────────────────┤
│ 响应：毫秒级  │ 响应：秒级          │ 响应：分钟级            │
│ 不离开阅读区  │ 打开右侧面板        │ 通知中心查看结果         │
│ 轻量、聚焦    │ 深度、对话式         │ 批量、自动化            │
└─────────────┴───────────────────┴───────────────────────┘
```

### 6.3 L1 即时层功能清单（悬浮窗内完成）

这些功能的特点是：**选中文字 → 即刻出结果 → 用户无需离开阅读上下文**。悬浮窗仅设三个按钮，保持极致简洁。

| 序号 | 功能 | 触发方式 | 输出形式 | 说明 |
|------|------|----------|----------|------|
| 1 | **引用** | 悬浮窗按钮 | 选中文字添加到右侧对话区「引用列表」 | 将选中文字作为上下文引用项，供后续对话使用 |
| 2 | **AI解释** | 悬浮窗按钮 | 悬浮窗下方展开解释面板 | 用通俗语言解释选中文字的含义，AI即时生成 |
| 3 | **AI翻译** | 悬浮窗按钮 | 悬浮窗下方展开翻译面板 | 翻译选中文字为目标语言（默认中英互译），AI即时生成 |

> **设计决策**：
> - L1层仅保留**引用、解释、翻译**三个核心按钮，极致简洁，降低认知负担。
> - **解释和翻译**采用独立的AI调用（如deepseek-chat），不附带文档全文上下文。响应快（1-3秒），结果直接在悬浮窗下方展开显示。
> - **引用**是纯本地操作，将选中文字添加到右侧对话面板的引用列表中，供后续L2深度对话时作为上下文片段使用。这是悬浮窗与右侧面板之间的桥梁。如需结合全文深度分析，用户使用L2层。

### 6.4 L2 对话层功能清单（右侧AI面板）

这些功能的特点是：**需要结合文档全文上下文、支持多轮对话、用户可能需要追问和深入**。

| 序号 | 功能 | 典型用户问法 | 输出特点 |
|------|------|-------------|----------|
| 1 | **全文问答** | "文档的核心结论是什么？" | 基于全文检索，标注来源章节 |
| 2 | **深度分析** | "分析第三章的实验方法是否合理" | 引用具体段落，多角度分析 |
| 3 | **跨章节对比** | "第二章和第五章的观点有什么矛盾？" | 跨章节检索+对比表格 |
| 4 | **概念关系梳理** | "文档中A和B是什么关系？" | 关系图描述+引用出处 |
| 5 | **写作辅助** | "帮我把这段改写得更学术化" | 输出改写版本+修改说明 |
| 6 | **引用生成** | "把这段内容生成APA格式引用" | 标准引用格式输出 |
| 7 | **阅读理解检测** | "根据文档内容，以下说法正确的是？" | 生成判断题/选择题 |
| 8 | **数据提取** | "把文档中所有的实验数据整理成表格" | 结构化表格输出 |

> **设计决策**：L2层的每次提问都携带**文档全文作为System Prompt上下文**（详见第十章提示词工程），确保AI回答始终基于文档内容，而非凭空生成。

### 6.5 L3 异步层功能清单（后台任务）

这些功能的特点是：**处理时间较长（10秒以上），适合后台执行，完成后通知用户查看**。

| 序号 | 功能 | 触发方式 | 输出位置 | 说明 |
|------|------|----------|----------|------|
| 1 | **全文摘要生成** | 打开文档自动触发 / 手动触发 | 通知 + 对话区首条消息 | 生成500-1000字的文档综合摘要 |
| 2 | **智能目录提取** | 文档导入时自动触发 | 左侧目录栏 | 对无目录文档自动生成章节结构 |
| 3 | **术语表生成** | 手动触发（快捷操作按钮） | 对话区 | 提取文档关键术语并解释 |
| 4 | **阅读建议** | 打开文档自动触发 | 对话区首条欢迎消息 | "这篇文档建议重点关注第X章..." |
| 5 | **多文档对比** | 手动触发 | 对话区 | 同时分析多个文档的异同（未来） |

### 6.6 功能矩阵：用户场景 × AI功能

| 用户场景 | 推荐L1功能 | 推荐L2功能 | 推荐L3功能 |
|----------|-----------|-----------|-----------|
| 读英文论文遇到生词 | 翻译 | - | - |
| 读一段难懂的技术描述 | 解释 | 深度分析 | - |
| 想快速了解文档讲了什么 | - | 全文问答 | 全文摘要生成 |
| 写论文需要引用 | 引用（收集段落） | 深度分析、全文问答 | - |
| 复习考试 | 引用（标记重点） | 阅读理解检测 | 术语表生成 |
| 对比两篇论文 | 引用（收集关键段） | 跨章节对比 | 多文档对比 |
| 翻译整段外文内容 | 翻译 | 全文问答（追问翻译问题） | - |

## 七、选中文字悬浮窗详细设计

### 7.1 触发机制

```
选中文字 → 松开鼠标 → 延迟200ms → 检查选中文本长度 ≥ 2 → 显示悬浮窗
                                                    ↓
                                          长度 < 2 → 不显示
```

**设计决策**：
- **200ms延迟**：避免快速划选时频繁弹出，减少视觉干扰
- **最小选中长度**：至少选中2个字符才触发（避免单击误触）
- **消失逻辑**：
  1. 点击悬浮窗及其展开面板外部任意区域
  2. 按下Esc键
  3. 开始新的文字选择（旧悬浮窗消失，新选中等200ms后弹新悬浮窗）
  4. 滚动文档
  5. 10秒无操作自动消失（可配置，比之前更长因为用户可能在看解释内容）

### 7.2 悬浮窗视觉设计

#### 初始状态（仅三个按钮）

```
                     选中文字区域
                    ┌──────────┐
                    │ 这是一段  │
                    │ 被选中的  │
                    │ 文字内容  │
                    └──────────┘
                         ▲ 尖角（8px，指向选中区域中心）
              ┌──────────────────────┐
              │  💬引用  💡解释  🔍翻译  │
              └──────────────────────┘
```

**布局规范**：
- 横向排列，三个按钮，每个按钮 = 图标（14px）+ 文字标签
- 按钮间距 4px，整体内边距 6px 12px
- 圆角卡片样式（radius 8px），白色背景 + `shadow-md` 阴影
- 尖角（8px 小三角）指向选中文字区域的中心
- 暗色主题下为深色背景（`--bg-card` dark）

**自适应定位**：
- 默认显示在选中文字**上方**（距选中区域上边缘8px）
- 如果上方空间不足（距窗口顶部 < 70px），自动切换为**下方**显示（尖角方向也翻转）
- 水平方向居中于选中文字，但不超出窗口左右边界（保留16px安全边距）

### 7.3 按钮尺寸与间距

```
┌─────────────────────────────────────────┐
│  ┌──────────┐ ┌──────────┐ ┌──────────┐│
│  │ 💬 引用   │ │ 💡 解释   │ │ 🔍 翻译   ││
│  └──────────┘ └──────────┘ └──────────┘│
│   ↑ 74px      ↑ 74px      ↑ 74px      │
└─────────────────────────────────────────┘
        ← 总宽度约 250px →
```

- 每个按钮宽度：74px，高度：36px
- 按钮圆角 6px，hover 时背景变为 `--bg-hover`
- 当前激活按钮（解释/翻译正在加载或已展示时）背景色变为 `--primary-light`

### 7.4 悬浮窗各按钮行为

| 按钮 | 点击后行为 | 视觉反馈 |
|------|-----------|----------|
| 💬 **引用** | 选中文字 + 所在页码/章节信息 → 添加到右侧对话面板「引用列表」 | 按钮短闪绿色 → Toast提示"已添加到引用列表" |
| 💡 **解释** | 调用AI解释选中文字 → 在悬浮窗下方展开解释面板显示结果 | 按钮变激活态 → 加载动画 → 展开解释面板 |
| 🔍 **翻译** | 调用AI翻译选中文字 → 在悬浮窗下方展开翻译面板显示结果 | 按钮变激活态 → 加载动画 → 展开翻译面板 |

> **互斥规则**：解释面板和翻译面板不能同时展示。点击「解释」时如果翻译面板已展开，则先收起翻译面板再展开解释面板；反之亦然。

```
    │     - 当前页码
    │     - 所属章节（如有）
    │     - 选中时间戳
    │
    ├─ 2. 添加到 AppState 当前 ChatSession 的引用列表
    │
    ├─ 3. 视觉反馈
    │     - 💬 引用按钮短暂变色（绿色闪烁 300ms）
    │     - Toast提示："已添加到引用列表"
    │
    └─ 4. 右侧对话面板
          - 引用列表更新，新引用项出现在列表顶部
          - 如果面板折叠 → 不自动展开（仅在引用区域显示数字角标）
          - 如果面板展开 → 新引用项高亮进入动画
```

### 7.7 悬浮窗与现有右键菜单的共存策略

- 悬浮窗和右键菜单**互斥展示**：右键时隐藏悬浮窗，显示右键菜单
- 右键菜单功能保持：复制 / AI提问 / 标注（原有功能不变）
- 悬浮窗是**高频快捷路径**（无需右键，松手即出），右键菜单是**低频备选路径**
- 右键菜单的「AI提问」行为不变：填充到右侧对话输入框并发送

## 八、右侧对话面板中的「引用列表」设计

### 8.1 引用列表的位置与布局

引用列表位于右侧AI对话面板**输入框上方、对话消息区下方**，是一个可折叠的区域。

```
┌─────────────────────────────────┐
│  AI 助手                    ✕  │  ← 面板标题栏

│                                 │
│  [对话消息区域]                   │  ← 可滚动
│                                 │
│  Turn 1: User                   │
│  Turn 1: AI                     │
│  Turn 2: User                   │
│  Turn 2: AI (思考中...)          │
│                                 │
├─────────────────────────────────┤
│  引用列表 (3)            ▲ ▼ 折叠│  ← 引用列表区域（可折叠）
│  ┌───────────────────────────┐  │
│  │ P12  "这是一段被选中的文字"  ✕│  │  ← 每条引用项
│  ├───────────────────────────┤  │
│  │ P5   "另一个引用的段落..."  ✕│  │
│  ├───────────────────────────┤  │
│  │ P12  "第三段引用内容..."   ✕│  │
│  └───────────────────────────┘  │
│  ┌───────────────────────────┐  │
│  │ 💡 AI解释: "这段话的核心..."✕│  │  ← 来自悬浮窗解释的引用
│  └───────────────────────────┘  │
├─────────────────────────────────┤
│  [输入框]                   [发送]│  ← 输入区域
└─────────────────────────────────┘
```

### 8.2 引用项数据结构

```javascript
// 每条引用项
{
  id: "ref-{timestamp}",
  type: "selection" | "explanation" | "translation",
  // selection = 直接引用选中文字
  // explanation = 来自悬浮窗"解释"的AI结果
  // translation = 来自悬浮窗"翻译"的AI结果

  text: "选中的原始文字内容",
  aiContent: "AI解释或翻译的结果文字",  // type为explanation/translation时有值

  pageNumber: 12,
  chapterTitle: "第三章 实验方法",       // 可选

}
```

### 8.3 引用列表的交互行为

| 操作 | 行为 |
|------|------|
| 点击引用项 | 阅读区跳转到对应页面，该段文字短暂高亮闪烁 |
| 点击 ✕ 删除 | 删除单条引用，其他引用项保持 |
| 点击「清空」 | 弹窗确认 → 清空所有引用项 |
| 点击折叠箭头 | 引用列表折叠为一行（仅显示"引用列表 (N)"） |
| 发送消息时 | 所有当前引用项自动作为消息的上下文附件发送给AI |
| 切换对话 | 引用列表**清空**（引用属于当前对话的临时上下文） |
| 切换文档 | 引用列表**清空** |

### 8.4 发送消息时的引用处理

当用户在输入框中输入问题并点击发送时，引用列表中的内容作为消息的一部分发送：

```

{
  role: "user",
  content: "请结合这些引用内容分析作者的观点",
  references: [                    ← 当前引用列表中的所有项
    {
      type: "selection",
      text: "这是一段被选中的文字",
      pageNumber: 12,
      chapterTitle: "第三章 实验方法"
    },
    {
      type: "explanation",
      text: "原始文字",
      aiContent: "AI解释内容...",
      pageNumber: 5
    }
  ]
}
```

**在System Prompt中的处理**（详见第十章）：

用户引用内容会以特殊格式注入到当前轮次的 User Message 中：

```
【用户引用的文档片段】

---
```

> **设计意图**：引用列表让用户在阅读过程中"收集"感兴趣的片段，然后在对话中一次性基于多个片段提问。这比传统的"选中 → 直接提问 → 选中 → 再提问"模式更加灵活，用户可以先收集上下文，再统一提问。

## 九、悬浮窗与右侧AI对话面板的布局关系与设计原则

### 9.1 核心设计原则

#### 原则一：就近原则（Proximity）
> 悬浮窗 + 展开面板全部附着在选中文字旁，用户视线移动范围 ≤ 300px。适用于「看一眼就能得到答案」的场景。

- 选中文字 → 悬浮窗出现在文字上方 → 解释/翻译面板紧接悬浮窗下方展开
- 用户视线移动路径：文字 → 悬浮窗（~50px） → 展开面板（+220px max），全程在阅读区域内

#### 原则二：渐进式披露（Progressive Disclosure）
> L1悬浮窗只做三件事（引用/解释/翻译），深度对话交给L2右侧面板。

```
选中文字
   │
   ├─→ 引用 → 添加到右侧面板引用列表 → 积累上下文 → L2深度对话
   ├─→ 解释 → 悬浮窗展开，就地展示AI结果 → 不满意？→ 底部的"引用到对话"→ L2追问
   └─→ 翻译 → 悬浮窗展开，就地展示AI结果 → 不满意？→ 底部的"引用到对话"→ L2追问
```

#### 原则三：不遮挡原则（Non-Obstruction）
- 悬浮窗在选中文字上方/下方自适应
- 展开面板紧接悬浮窗，高度最大 220px
- 点击面板外部、按Esc、滚动文档 → 全部消失
- 面板内容超长时 Y 轴滚动

#### 原则四：悬浮窗轻量、面板深度（Lightweight vs. Deep）
> 悬浮窗 = 阅读中的"即看即走"，右侧面板 = 阅读后的"深度分析"。

| 维度 | 悬浮窗 + 展开面板 | 右侧AI面板 |
|------|------------------|-----------|
| 定位 | 快速查阅、收集片段 | 深度对话、综合提问 |
| 交互模式 | 点击 → 看结果 → 可能引用 → 关闭 | 多轮对话 |
| AI上下文 | 仅选中文字 | 全文 + 引用片段 + 对话历史 |
| 持久性 | 阅后即焚（展开面板内容不保存） | 永久保存到对话历史 |
| 引用能力 | 可"引用"到右侧面板 | 自动携带当前引用列表 |
| 使用频率 | 高频（15+次/阅读） | 中频（3-5次/阅读） |

### 9.2 悬浮窗与右侧面板的交互流程

```
                    ┌──────────────┐
                    │  用户选中文字  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  悬浮窗弹出    │
                    │ 💬引用 💡解释  │
                    │ 🔍翻译        │
                    └──┬───┬───┬──┘
                       │   │   │
           ┌───────────┘   │   └──────────┐
           ▼               ▼              ▼
    ┌────────────┐  ┌────────────┐  ┌────────────┐
    │ 💬 引用      │  │ 💡 解释      │  │ 🔍 翻译      │
    │ 添加到右侧   │  │ 展开解释面板 │  │ 展开翻译面板 │
    │ 引用列表     │  │ AI即时生成  │  │ AI即时生成  │
    └─────┬──────┘  └──────┬─────┘  └──────┬─────┘
          │               │               │
          │        ┌──────▼─────┐  ┌──────▼─────┐
          │        │ 用户查看结果 │  │ 用户查看结果 │
          │        └──────┬─────┘  └──────┬─────┘
          │               │               │
```

### 10.2 实体对应关系总结

| 关系 | 类型 | 说明 |
|------|------|------|
| Document → ChatSession | **1 : N** | 一个文档可以有多个独立对话（如"第3章分析"、"论文对比"） |
| ChatSession → Message | **1 : N** | 一个对话包含多轮问答 |
| ChatSession → QuotedItem | **1 : N** | 一个对话有一个引用列表（临时上下文，切换对话清空） |
| Document → DocumentContent | **1 : 1** | 每个文档有且仅有一份解析后的全文内容 |
| Document → Highlight | **1 : N** | 一个文档可以有多个高亮标注 |
| Selection → QuotedItem | **瞬态创建** | 用户选中文字后点「引用」，创建引用列表项 |
| Selection → 任何实体 | **瞬态** | 选中文字本身不持久化，仅作为触发动作的临时数据 |
| APIKeyConfig | **全局单例** | 整个应用只有一套API配置（未来可扩展多provider） |

### 10.3 数据存储方案

| 数据 | 存储位置 | 原因 |
|------|----------|------|
| 文档元数据（Document） | IndexedDB | 结构化数据、支持索引查询 |
| 文档全文内容（DocumentContent） | IndexedDB | 体积可能较大（单文档可达数MB） |
| 对话记录（ChatSession + Message） | IndexedDB | 需要按文档ID查询、按时间排序 |
| 高亮标注（Highlight） | IndexedDB | 与文档关联、需要持久化 |
| 引用列表（QuotedItem） | **内存（AppState）** | 临时上下文，切换对话或文档时清空 |
| 用户设置（主题、语言等） | localStorage | 小体量键值对、全局访问 |
| API Key配置 | localStorage（混淆编码） | 小体量、需要在HTTP请求前读取 |
| 阅读位置 | localStorage | 快速读写、每个文档一个键 |
| PDF Blob URL | 内存（URL.createObjectURL） | 仅会话期间有效 |

> **为什么不使用localStorage存储全部数据？** localStorage有5MB限制，单个长文档的全文内容就可能超过。IndexedDB提供了更大的存储空间（通常数百MB甚至更多）和更好的查询能力。
> 
> **为什么引用列表只存内存？** 引用列表是"阅读时收集的临时上下文"，用户在发送消息后引用即被消费。它属于当前对话的临时工作区，不应污染持久化存储。用户可以通过对话历史回看之前发送的引用内容。

### 10.4 Message 的 source 字段设计

每条AI回复的 Message 带有 `source` 字段，用于标注回答基于文档的哪些位置：

```javascript
// source 字段的数据结构
{
  type: "chapter" | "page" | "selection" | "fulltext",
  refs: [
    { chapter: "第三章 实验方法", page: 12 },
    { chapter: "第五章 结论", page: 23 }
  ],
  confidence: 0.85  // AI对来源相关性的置信度
}
```

**前端展示**：
- 在AI回复开头显示 `📍 基于第三章 实验方法（第12页）、第五章 结论（第23页）`
- 点击来源标签可跳转到对应页面
- 如果 `confidence < 0.5`，显示 `⚠️ 以下回答部分基于通用知识`

### 11.1 整体架构：三层提示词体系

```
┌────────────────────────────────────────────────────┐
│                   System Prompt                     │
│  ┌──────────────────────────────────────────────┐  │
│  │ 1. 角色设定（Role Card）                       │  │
│  │    "你是一个专业的阅读助手..."                  │  │
│  ├──────────────────────────────────────────────┤  │
│  │ 2. 文档全文上下文（Document Context）            │  │
│  │    "以下是你需要帮助用户理解的文档全文："         │  │
│  │    [文档完整文本，按章节组织]                    │  │
│  ├──────────────────────────────────────────────┤  │
│  │ 3. 回答规则（Response Rules）                   │  │
│  │    "回答时请标注来源章节和页码..."               │  │
│  └──────────────────────────────────────────────┘  │
├────────────────────────────────────────────────────┤
│                   User Messages                     │
│  每轮对话携带：当前问题 + 可选选中文字引用            │
├────────────────────────────────────────────────────┤
│                Conversation History                  │
│  最近N轮对话记录（受Token限制）                       │
└────────────────────────────────────────────────────┘
```

### 11.2 System Prompt 详细设计

#### 角色设定（固定部分，约200 tokens）
```
你是一个专业的AI阅读助手，帮助用户理解和分析文档内容。
你的回答应该：
- 准确：严格基于文档内容，不编造信息
- 清晰：使用通俗易懂的语言解释复杂概念
- 结构化：使用列表、分点等方式组织回答
- 可追溯：每个观点都标注在文档中的来源位置
```

#### 文档上下文（动态部分，受Token限制）
```
【文档标题】：{document.name}
【文档类型】：{document.type}
【章节结构】：
  第一章：引言（第1-3页）
  第二章：相关工作（第4-8页）
  ...

【文档全文】：
{第一章 引言}
...（章节完整文本）...
{第二章 相关工作}
...（章节完整文本）...
```

#### 回答规则（固定部分，约150 tokens）
```
回答规则：
1. 每个观点后标注来源，格式："（见第X章 XXX，第Y页）"
2. 如果文档中找不到相关信息，明确说"文档中未提及此内容"
3. 如果需要补充通用知识，先声明"以下为通用知识补充："
4. 回答长度：{根据用户设置动态调整}
5. 使用Markdown格式化回答
```

### 11.3 Token预算与上下文窗口管理

DeepSeek-V3 的上下文窗口为 64K tokens。需要在 System Prompt（文档全文）和 Conversation History 之间做平衡。

#### Token分配策略

| 场景 | 文档Token | 对话历史Token | 回答预留Token | 说明 |
|------|-----------|--------------|--------------|------|
| 短文档（< 10K） | 实际大小 | 剩余全部（最多50K） | 4K | 文档全文 + 长对话历史 |
| 中文档（10K-40K） | 实际大小 | 剩余（最多20K） | 4K | 文档全文 + 适量历史 |
| 长文档（> 40K） | 40K（截断/压缩） | 16K | 4K | 智能截取 + 保留最近对话 |

> **当前阶段先简化**：先限制支持文档大小不超过DeepSeek上下文窗口，超标时在前端直接截断并提示用户。后续版本再实现RAG分块检索。

#### 长文档处理策略（后续版本）

```
长文档 → 分块（按章节/1000字） → 向量化 → 存储到本地向量数据库
                                           ↓
用户提问 → 问题向量化 → 检索Top-K相关块 → 拼接为上下文
```

**当前版本（V1.0）采用简化方案**：
1. 文档文本直接拼接在System Prompt中
2. 如果文档Token超过限制（如40K），取前40K字符 + 提示"文档过长，仅分析前40K字符内容"
3. 章节结构始终完整保留（章节标题和页码范围很小），确保AI知道文档整体结构

### 11.4 记忆管理模型

#### 短期记忆（Short-term Memory）—— 当前对话窗口
- **内容**：最近N轮对话的User + AI消息对
- **范围**：仅限当前 ChatSession
- **管理策略**：Token计数 + FIFO淘汰。超出预算时，最早的消息对出窗口
- **可视化**：对话区顶部显示Token使用进度条 "上下文使用：18K / 64K"

```
┌──────────────────────────────────────────────┐
│  System Prompt (文档全文)                     │  ← 固定，不淘汰
│  ─────────────────────────────────────────── │
│  Turn 1: User + AI                          │
│  Turn 2: User + AI                          │  ← 对话历史窗口
│  Turn 3: User + AI (当前)                    │     动态滑动
│  Turn 4: User (最新)                         │
└──────────────────────────────────────────────┘
```

#### 长期记忆（Long-term Memory）—— 文档级别
- **全文摘要**：L3异步生成的500-1000字摘要，缓存在 DocumentContent.summary
- **术语表**：DocumentContent.termList，提取的关键术语及定义
- **用途**：在新对话开始时，将摘要和术语表注入System Prompt（节省Token的同时保持对文档的整体理解）

#### 对话记忆（Conversation Memory）—— 会话级别
- 每个 ChatSession 的完整消息历史**永久存储在IndexedDB**
- 加载历史对话时，完整恢复所有消息
- 重新进入对话时，仅最近N条放入Context Window，但用户可以向上滚动查看完整历史

### 11.5 不同AI功能的提示词策略

| 功能 | 层级 | System Prompt | 是否携带文档全文 |
|------|------|--------------|----------------|
| AI解释（悬浮窗） | L1 | 角色设定 + "请用通俗语言解释以下文字" | ❌ 不携带 |
| AI翻译（悬浮窗） | L1 | 角色设定 + "请翻译以下文字" | ❌ 不携带 |
| 引用列表（发送时） | L2 | 完整System Prompt + 引用片段注入User Message | ✅ 携带 |
| 全文问答（右侧面板） | L2 | 完整System Prompt | ✅ 携带 |
| 深度分析（右侧面板） | L2 | 完整System Prompt | ✅ 携带 |
| 跨章节对比（右侧面板） | L2 | 完整System Prompt | ✅ 携带 |
| 全文摘要（后台任务） | L3 | 角色设定 + "请总结以下文档" | ✅ 携带 |

> **关键设计决策**：
> - L1层（解释/翻译）不携带文档全文，这让L1的AI调用轻量、快速（响应时间1-3秒），适合高频使用。
> - L2层发送消息时，用户通过悬浮窗「引用」收集的片段会以特殊格式注入User Message，AI同时拥有文档全文 + 用户精选片段，回答更加精准。
> - 用户如果对L1的回答不满意，可以通过展开面板底部的「引用此解释/翻译到对话」将AI结果也加入引用列表，然后在右侧面板追问。

### 12.5 安全存储方案

```
存储流程：
API Key明文 → XOR混淆（用设备指纹作为key） → Base64编码 → localStorage

读取流程：
localStorage → Base64解码 → XOR解混淆 → 内存中使用（不写回任何地方）
```

**安全措施**：
- API Key 永远不在前端控制台日志中输出
- 发送到AI服务时通过HTTPS传输
- 设置页面离开后，输入框清空（仅保留混淆后的值）
- 不将API Key包含在任何错误报告/诊断信息中
